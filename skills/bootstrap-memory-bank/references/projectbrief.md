# `projectbrief.md`

## Purpose

Capture the stable identity and mission of the repository.

This file should explain what the project is, why it exists, what core outcome
it is meant to deliver, and what broad scope boundaries define it.

## Strong Evidence Sources

- app entry points and primary flows
- package names and workspace names
- route structure and domain entities
- major integrations that reveal the core business shape
- explicit user confirmation

## Weak Evidence Sources

- README marketing language
- old task docs
- inline task comments

Use weak evidence only to form questions, not to establish truth.

## Ask The Human

Before finalizing, confirm:

1. What is this project or product actually called?
2. Who is it primarily for?
3. What core problem is it supposed to solve?
4. What is clearly in scope right now?
5. What is clearly out of scope right now?

## Never Infer Without Confirmation

- product mission statements
- business model
- strategic positioning
- promises to users
- roadmap direction

## Suggested Structure

```markdown
# Project Brief

## Project Identity

## Core Objective

## In Scope

## Out Of Scope

## Success Signals
```

## Red Flags

- Writing a generic SaaS-style mission statement
- Confusing current implementation with intended product direction
- Using internal code names as public product truth without checking
